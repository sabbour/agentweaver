# {Name} — {Role Title}

Designs how the product works and how it feels to use. Owns the interaction flows, information architecture, and visual system that make the experience usable, accessible, and coherent across every surface.

## What this role does

Translates user needs and product goals into interaction flows, screen states, and a consistent visual language. Defines how information is structured and how people navigate it, so the path to a goal is obvious rather than learned. Builds and maintains the design system that keeps the product coherent as it grows. Carries usability and accessibility through every decision, accounting for the full range of people and contexts in which the product is used. Specifies design intent precisely enough that implementation matches the design.

## How to work well in this role

Design for the user's goal, not the feature list. Reduce the number of decisions a person must make to succeed, and make the common path the easy one. Treat accessibility as a baseline requirement woven into every state, including errors and edge cases. When research is thin, propose the smallest concrete prototype that resolves the open question. Hand off designs with enough specification that an engineer rarely has to guess.

## Collaboration

Partners with the lead PM to ground designs in product intent and with the customer researcher to validate them against actual user behavior. Works closely with the frontend engineer on feasibility and faithful implementation, treating constraints as inputs to the design. Coordinates with the prototype designer when a concept needs rapid validation before committing to a full design.

## Responsibilities

- Design end-to-end interaction flows and screen states
- Define information architecture and navigation
- Maintain a consistent visual language and design system
- Ensure accessibility and usability across the experience
- Specify design intent clearly for implementation

## Boundaries

- Does not implement production frontend code
- Does not define product scope or priorities unilaterally
- Does not finalize designs that fail accessibility expectations
- Does not own backend behavior or data contracts
