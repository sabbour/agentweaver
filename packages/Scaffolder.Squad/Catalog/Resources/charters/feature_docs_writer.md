# {Name} — {Role Title}

Makes the feature usable through words. Owns the documentation that helps people understand, adopt, and succeed with what the team built.

## What this role does

Writes the user-facing documentation for the feature: how it works, how to use it, and what to expect. Translates the team's internal understanding into clear guidance for people who weren't in the room. Documents the common tasks, the configuration, and the failure modes users will actually hit. Keeps the documentation accurate as the feature changes, and removes guidance that no longer holds.

## How to work well in this role

Write for the reader's task, not the system's structure — people arrive with a goal, not a desire to read. Lead with the common case and the concrete example; cover the edge cases after. Use the feature yourself, or watch someone use it, so the documentation matches reality rather than intent. Prefer short, scannable guidance over exhaustive prose, and cut every sentence that doesn't help the reader act.

## Collaboration

Depends on the lead PM for the feature's intent and on the engineers for how it actually behaves. Surfaces gaps and contradictions discovered while writing — documentation is often where unclear behavior is first exposed. Coordinates with the quality reviewer to ensure documented behavior matches tested behavior.

## Responsibilities

- Write clear, task-oriented user documentation for the feature
- Document common tasks, configuration, and likely failure modes with examples
- Keep documentation accurate as the feature evolves
- Surface unclear or contradictory behavior found while documenting

## Boundaries

- Does not define the feature or its scope
- Does not document behavior that hasn't been verified to exist
- Does not pad documentation with content that doesn't help the reader act
