### Decisions — Coordinator (feature 009)

#### 2026-06-19: Path A (full coordinator pickup) for backlog Ready→Run promotion
**By:** Coordinator (user-chosen, Path A selected after Tank's design revision)
**Branch:** `009-backlog-kanban-board`
**Status:** Approved

Each Ready backlog item is promoted to a COORDINATOR run via `CoordinatorRunService`, reusing the existing coordinator pipeline (FR-021). No new run type or executor is introduced; the existing decompose→dispatch coordinator path handles the item end-to-end.

**Rationale:** Path A reuses all existing coordinator governance, RAI, and audit surface. The alternative (direct worker dispatch) would have bypassed decomposition and plan approval, violating constitution principle I.

#### 2026-06-19: New process-wide CoordinatorHeartbeatService for periodic pickup
**By:** Morpheus
**Branch:** `009-backlog-kanban-board`
**Status:** Approved

A new `CoordinatorHeartbeatService` (PeriodicTimer, 10s default interval, `Coordinator:HeartbeatEnabled` feature flag) was added because no periodic heartbeat existed prior to feature 009. "Active coordinator for a project" is defined as: `project.State == Active` AND a workspace is available. `CoordinatorPickupService` iterates all active projects on each tick and promotes Ready items via `CoordinatorRunService.StartReservedCoordinatorRunAsync`.

**Rationale:** Pickup requires a background loop independent of HTTP request lifecycle. A dedicated hosted service with a feature flag keeps it opt-in and testable without side effects in unit tests.

#### 2026-06-19: Exactly-once pickup via a single SQLite transaction with origin marker
**By:** Tank + Morpheus (governance fix)
**Branch:** `009-backlog-kanban-board`
**Status:** Approved

Atomic pickup is enforced by a single SQLite transaction in `SqliteBacklogTaskStore.TryClaimAndReserveCoordinatorRunAsync`: claim `Ready → Claimed` gated on `run_id IS NULL`, then insert the `runs` row with `origin = 'backlog_pickup'` and the `workflow_runs` envelope atomically. The durable `runs.origin` marker gates unattended recovery: only `origin = 'backlog_pickup'` runs are eligible for auto-confirm; interactive runs are never auto-confirmed. `confirmedBy` is set to `task.CapturedBy`, preserving a named human accountable for the task.

**Rationale:** Without the atomic claim+insert, two concurrent pickup ticks could both promote the same Ready item. The origin marker prevents the unattended-recovery path from auto-confirming interactive runs, which would violate the human-in-the-loop governance requirement.

#### 2026-06-19: Board columns dynamically derived from CoordinatorGraphDescriptor
**By:** Tank
**Branch:** `009-backlog-kanban-board`
**Status:** Approved

Board columns are built from `CoordinatorGraphDescriptor` at render time: Backlog, Ready, then each workflow stage in descriptor order, then Done. No column list is hardcoded in the frontend or backend. The FR-019 fallback renders an intake-only board with a workflow-unavailable indicator when the descriptor cannot be resolved (e.g., no coordinator configured for the project). `IWorkflowStageProjector` was extracted as an interface to make the fallback path unit-testable.

**Rationale:** Hardcoding columns would require a code change for any workflow topology change. Dynamic derivation ensures the board always reflects the actual configured workflow graph.

#### 2026-06-19: Board live updates via 3s polling of GET /board
**By:** Trinity
**Branch:** `009-backlog-kanban-board`
**Status:** Approved

The web Kanban board polls `GET /api/projects/{id}/board` every 3 seconds via a `useBoard` hook. Mutations (capture task, promote to Ready, drag-drop) trigger an immediate refetch. A project-SSE push upgrade was considered and deferred to a future feature.

**Rationale:** Polling is simple, stateless, and consistent with the existing run-list refresh pattern in the UI. SSE upgrade adds complexity (connection management, reconnect logic) that is not justified by the current latency requirement.

## 2026-06-19: Feature 009 — Retrigger Failed Run

### Decisions — Tank (Backend)

#### 2026-06-19: POST /api/runs/{id}/retry — fresh run, never mutate
**By:** Tank (backend)
**Branch:** 009-backlog-kanban-board
**Status:** Approved

A retry mints a new `RunId` reconstructed from the failed run's persisted inputs (Task, Project, SubmittingUser, Repo, Branch, Model, Origin) and links back via the new `Run.RetriedFrom`. The source run is never touched — it stays Failed/MergeFailed with its history intact.

Eligible source states: `Failed` and `MergeFailed` → 201 Created. `Declined` → 409 (a human decision is not silently re-litigated). Child runs (`ParentRunId != null`) → 409 (retried through their coordinator parent). Every other state (terminal-success, in-flight) → 409 `{"error":"run_not_retryable","status":"<x>"}`.

Soft retry cap = 3: the endpoint walks the `retried_from` provenance chain; if source run's chain depth >= 3, returns 409 `{"error":"retry_limit_reached"}`.

Three run shapes, three start seams: interactive coordinator reuses `StartCoordinatorRunAsync` with new optional `retriedFrom` arg; backlog-pickup coordinator uses new `StartRetriedPickupCoordinatorRunAsync` (no claim+reserve, backlog task already Claimed, preserves Origin=BacklogPickup + SubmittingUser); regular project run rebuilds via `RunOrchestrator.StartRunAsync`.

Outcome spec / work plan are re-drafted under the new run_id (the old spec may itself be the cause of failure).

Auth: owner-scoped, 401/403/404 mirroring sibling mutations. Response: `201 Created`, `Location: /api/runs/{newId}`, body `RetryRunResponse { run_id, retried_from, status }` (snake_case). `retried_from` also exposed on `RunResponse` and `RunCardDto`.

Files: `Run.cs` (RetriedFrom), `SqliteDb.cs` (migration), `SqliteRunStore.cs` (write/read), `CoordinatorRunService.cs`, `RunEndpoints.cs`, `BoardProjectionService.cs`, `Dtos.cs`, `RunRetryTests.cs` (13 cases). Backend suite: 701 passed, 2 skipped, 0 failed. Reference: `tank-retrigger`.

### 2026-06-19: Phase 2 — project-aggregate per-agent work-queue rollup on the board DTO (agent_queues)
**Source:** .squad\decisions\inbox\tank-agent-queues-p2.md


- Branch: 009-backlog-kanban-board
- Author: Tank (backend)
- Requested by: @sabbour
- Status: Implemented + verified

#### Goal
Feed the homepage "Agents" rail with a project-aggregate per-persona work-queue rollup, additively on
the existing board response (GET /api/projects/{projectId}/board). Trinity builds the web side in
parallel against the locked snake_case contract.

#### Contract (locked, implemented exactly)
New top-level `agent_queues: AgentQueueDto[]` on the board response, sibling to `columns`:
```
AgentQueueDto = {
  agent_name: string;
  active: number; queued: number; blocked: number; done: number;
  run_ids: string[];      // coordinator run ids contributing >=1 subtask for this agent
  sample_titles: string[];// up to 3, preferring queued+active titles
}
```
Empty (never null) when there is no active subtask work.

#### Server-side status -> bucket mapping (mirrors apps/web/src/api/agentQueues.ts exactly)
Source of truth: `Subtask.Status` (apps/Agentweaver.Api/Memory/Subtask.cs), whose server-side values
are: `pending | dispatched | running | rai_flagged | assemble_ready | completed | failed`.

| server-side subtask status | bucket  |
|----------------------------|---------|
| dispatched                 | active  |
| running                    | active  |
| in_progress (*)            | active  |
| pending                    | queued  |
| (any unrecognized status)  | queued  |
| failed                     | blocked |
| rai_flagged                | blocked |
| completed                  | done    |
| assemble_ready             | done    |
| merged (*)                 | done    |

(*) `in_progress` and `merged` are not emitted for subtasks today but are kept for forward-parity with
the web mapping. Match is case-insensitive (ToLowerInvariant), identical to the web switch.

#### Which runs are aggregated
The ACTIVE (non-terminal) coordinator runs backing the board: during the existing run-bucketing loop
each run already gets a stage id via `WorkflowStageProjector.CoordinatorRunToStageId`; runs whose
stage id != `TerminalStageId` are collected as `activeRunIds`. Historical/merged-away runs in the
terminal "Done" column are excluded so the rail reflects current in-flight work only. (Verified by a
test that a Merged run's completed subtask does NOT inflate the rollup.)

#### Performance
One additional batched query, no N+1: `WorkPlans` filtered to the active run ids are joined to
`Subtasks` and projected to (CoordinatorRunId, AssignedAgent, Status, Title) in a single round trip
(`BuildAgentQueuesAsync`). The board's existing per-run lightweight stage projection
(`GetWorkPlanStagesAsync`) is unchanged. So the homepage board load adds exactly one EF query (skipped
entirely when there are no active runs). Subtask rows were NOT already in the projection's working set
(the stage projection reads only WorkPlan Status/AssemblyStage), so loading them is the minimum needed.

#### Aggregation details
- Persona key = `Subtask.AssignedAgent`; blank/whitespace falls back to "Unassigned" (mirrors web).
- `run_ids`: distinct coordinator run ids contributing a subtask for the agent (sorted Ordinal for
  determinism).
- `sample_titles`: titles from queued+active subtasks only, in encounter order, capped at 3.
- Ordering: most-loaded first via `active*4 + queued*2 + blocked` descending (mirrors the web sort),
  with an Ordinal agent-name tie-break for stable output.

#### Files changed
- `apps/Agentweaver.Api/Contracts/Dtos.cs` — added `AgentQueueDto` (snake_case) and the required
  `agent_queues` field on `BoardDto`. Existing columns/cards unchanged (purely additive).
- `apps/Agentweaver.Api/Runs/BoardProjectionService.cs` — collect `activeRunIds` in the existing loop;

### 2026-06-19: Fix 403 regression on backlog-pickup coordinator runs (GitHub-identity-aware ownership)
**Source:** .squad\decisions\inbox\tank-pickup-owner-403.md


- Branch: 009-backlog-kanban-board
- Author: Tank (backend)
- Requested by: @sabbour
- Status: Implemented + verified

#### Problem
The earlier CapturedBy=GitHub-login change made `BacklogTask.CapturedBy` (and therefore a
pickup run's `Run.SubmittingUser`, set at `CoordinatorPickupService.cs:65`) the signed-in GitHub
login (e.g. "sabbour"). But ownership was checked in `EndpointHelpers.IsOwner` (and several inline
sites) as `caller.User == run.SubmittingUser`, where `caller.User` is the API-key principal
(`Auth:User`, e.g. "local-developer"). For pickup runs the two never matched, so EVERY owner-scoped
endpoint (GET /api/runs/{id}, /outcome-spec, /work-plan, /graph, /children, assembly/*) returned 403
and the Orchestration page showed "API error 403". Interactive runs were unaffected because they
store `SubmittingUser = caller.User`.

#### Decision — Approach A: GitHub-identity-aware ownership (chosen)
Enrich the per-request caller once with the resolved signed-in GitHub login, and treat a run as owned
when its `SubmittingUser` matches EITHER the API-key principal OR that GitHub login.

Why A over the two-field alternative:
- `IGitHubTokenStore.GetIdentityAsync`/`GetAsync` resolve identity from LOCAL storage only (file /
  OS credential store / in-memory) — verified across all three implementations. No network on the
  hot path. The GitHub network call happens only during device-flow sign-in.
- `IGitHubTokenStore` and `IGitHubTokenScopeProvider` are singletons; resolution is cheap.
- Keeps `SubmittingUser` = the accountable GitHub login (Principle IX) with NO new DTO/display field
  and NO web change required (Trinity owns CoordinatorRunPage.tsx — untouched).
- Interactive runs keep working via the principal branch.
- Resolution is best-effort: wrapped in try/catch, never throws, never blocks the request; on any
  failure the GitHub login is simply null (falls back to principal-only ownership).

Note on scope model: `FixedInstallationScopeProvider` maps every caller to the single Installation
scope, so in the local single-installation model all authenticated callers share the signed-in
identity. This is acceptable for the local backend and is exactly what lets the signed-in human view
their picked-up runs.

#### Changes
- `apps/Agentweaver.Api/Security/ApiKeyAuthMiddleware.cs`
  - `CallerContext` gains `string? GitHubLogin` and `bool Owns(string? ownerUser)` (Ordinal, null-safe;
    matches `User` OR `GitHubLogin`).
  - `InvokeAsync` now takes `IGitHubTokenStore` + `IGitHubTokenScopeProvider` (method injection) and
    resolves the signed-in login once per request via a try/catch-guarded helper; stashes it on the caller.
  - Added `using Agentweaver.Domain;`.
- `apps/Agentweaver.Api/Endpoints/EndpointHelpers.cs` — `IsOwner` now delegates to `caller.Owns(run.SubmittingUser)`.
  This fixes ALL endpoints that route through `IsOwner` (RunEndpoints + CoordinatorEndpoints /outcome-spec,
  /work-plan, /children, assembly/*).
- `apps/Agentweaver.Api/Endpoints/RunEndpoints.cs` — the inline owner comparisons now use `caller.Owns(...)`:
  DELETE (403), /stream owner checks (KEEP 404 info-hiding semantics), and the two pending-request IDOR
  defense-in-depth checks (`pendingEntry.OwnerUser`).
- `apps/Agentweaver.Api/Coordinator/AssemblyReviewGate.cs` — `TrySubmit` gains an optional
  `string? callerGitHubLogin` so its IDOR owner check is identity-aware too.
- `apps/Agentweaver.Api/Endpoints/CoordinatorEndpoints.cs` — passes `caller.GitHubLogin` to `TrySubmit`.

#### /stream 404 (not 403)
Same root cause. The stream endpoint deliberately returns 404 (not 403) on owner mismatch to hide run
existence. The underlying owner comparison was fixed to be identity-aware (so the signed-in owner is
recognized), while the 404 response on a genuine non-owner is preserved by design.

#### Tests
`tests/Agentweaver.Tests/Coordinator/PickupRunOwnershipTests.cs` (6 tests, hermetic — real
`ProjectsWebApplicationFactory` + in-memory token store seam, pickup-shaped run inserted directly via
the real `SqliteRunStore`, status Failed so no orchestration/network):
1. Signed-in GitHub user can GET a pickup run attributed to their login -> 200.
2. Signed-in GitHub user can open /stream for that run -> not 404 (identity-aware).
3. Signed-in user CANNOT view a run owned by a different login -> 403.
4. Interactive run owned by the API principal still resolves when signed out -> 200 (parity preserved).
5. Signed-out caller viewing a run attributed to a GitHub login -> 403 (no over-broad access).
6. Unauthenticated -> 401.
