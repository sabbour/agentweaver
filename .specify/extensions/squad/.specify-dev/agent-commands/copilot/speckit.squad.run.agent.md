# Squad Bridge: Run

Execute routed tasks by spawning Squad agent sessions. Reads the task list
and agent assignments produced by `/speckit.squad.route`, then launches
agents to do the work — creating branches, running sessions, and optionally
creating PRs.

This is the command that closes the loop: specs become plans, plans become
tasks, tasks get routed to agents, and this command makes agents execute.

## Prerequisites

- Tasks must be routed: run `/speckit.squad.route --update-tasks` first
- Squad CLI available: `squad --version`
- GitHub CLI authenticated: `gh auth status`

## User Input

$ARGUMENTS

## Steps

1. **Read tasks** from `specs/<id>/tasks.md`. Identify tasks that have squad
   routing metadata (comments like
   `<!-- squad:agent=backend-engineer tier=standard -->`).
   If no routed tasks found, tell the user to run
   `/speckit.squad.route --update-tasks` first and stop.

2. **Load Squad agents** from `.squad/agents/` and verify each assigned agent
   exists and has `status: active`.

3. **Load bridge config** from `.specify/extensions/squad/squad-config.yml`
   if it exists, otherwise use extension defaults.

4. **Parse arguments**:
   - No args: run all pending routed tasks
   - Task ID or title fragment (e.g., `task-03`): run only that task
   - `--dry-run`: show what would execute without starting sessions
   - `--retry`: re-run previously failed tasks only
   - `--retry <task-id>`: re-run a specific failed task

5. **If `--dry-run`**, print the execution plan and stop:

   ```
   Dry Run — would execute:
   ─────────────────────────────────────────────────────────
   Task                              Agent               Tier
   ─────────────────────────────────────────────────────────
   Design REST API endpoints         backend-engineer    full
   Implement JWT authentication      backend-engineer    standard
   Build login form component        frontend-engineer   standard
   Write unit tests for auth         qa-engineer         standard
   ─────────────────────────────────────────────────────────
   4 tasks | 3 agents | parallel limit: 3

   Run without --dry-run to execute.
   ```

6. **For each routed task** (up to `execution.parallel_limit` concurrently):

   a. **Create a working branch** using the configured pattern:
      ```bash
      git checkout -b squad/{agent}/{task-id}
      ```

   b. **Build agent prompt** — construct a detailed prompt that includes:
      - The task title and description from tasks.md
      - Relevant sections from the plan (the phase this task belongs to)
      - The agent's charter (role, capabilities from charter.md)
      - Conventions from the agent's history.md (if it exists)
      - Success criteria: what "done" looks like for this task

   c. **Launch the agent session**:
      ```bash
      gh copilot \
        -p "<agent-prompt>" \
        --agent squad \
        <copilot_flags from config>
      ```

   d. **Monitor session** — watch for completion or timeout. If
      `execution.session_timeout_minutes` is exceeded, mark as `timed_out`.

   e. **On completion**:
      - If `execution.auto_pr` is true, create a pull request:
        ```bash
        gh pr create \
          --title "Squad: <task-title>" \
          --body "## Task\n<task description>\n\n## Agent\n<agent name>\n\n## Context\nGenerated by /speckit.squad.run" \
          --base main
        ```
      - Record result (success/failure, duration, branch name)

7. **Print execution summary**:

   ```
   Execution Summary
   ─────────────────────────────────────────────────────────
   ✅ Design REST API endpoints      (backend-engineer, 12m)
   ✅ Implement JWT authentication   (backend-engineer, 8m)
   ✅ Build login form component     (frontend-engineer, 15m)
   ❌ Write unit tests for auth      (qa-engineer, failed)
   ─────────────────────────────────────────────────────────
   3/4 succeeded | 1 failed

   Branches created:
     squad/backend-engineer/task-01
     squad/backend-engineer/task-02
     squad/frontend-engineer/task-03
     squad/qa-engineer/task-04

   To retry failed tasks: /speckit.squad.run --retry
   To check status:       /speckit.squad.status
   ```

## Notes

- Each agent session runs on its own branch. This avoids merge conflicts
  between agents working in parallel.
- The `--dry-run` flag is recommended before first execution to verify the
  plan looks correct.
- Agent sessions inherit the agent's accumulated knowledge from `history.md`
  — agents get smarter across repeated runs.
- This command does not reorder tasks by dependency. Tasks are executed in
  the order they appear in tasks.md. If dependency ordering is important,
  arrange tasks accordingly before running, or run specific tasks/groups
  manually with task ID arguments.
- Execution results are not persisted to a state file by default. Use
  `/speckit.squad.status` after execution to review the overall picture.
